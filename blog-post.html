Mastering the Page Object Model (POM)

Piyush Baldaniya
• Oct 24, 2025 • 10 min read

"If your test scripts are littered with `driver.findElement(By.xpath("//div..."))`, you are not writing automation; you are writing a maintenance nightmare."

Imagine a login page. You have 50 test cases that need to log in. In a linear script ("Spaghetti Code"), you would write the locator for the username field 50 times. Now, imagine the developer changes the ID of that username field. You now have to find and replace that locator in 50 different places.

This is where the Page Object Model (POM) saves the day. It acts as an "Object Repository," separating your page elements (locators) from your tests (logic).

The "Old Way" vs. POM

In the old way, logic and data are mixed. This is hard to read and harder to debug. 

public void testLogin() {
    // Spaghetti Code: Hardcoded locators inside the test
    driver.findElement(By.id("username")).sendKeys("admin");
    driver.findElement(By.id("password")).sendKeys("12345");
    driver.findElement(By.cssSelector("btn-login")).click();
}

Enter Page Factory

Selenium provides a built-in support class called PageFactory to initialize web elements. We use the @FindBy annotation to define elements nicely at the top of the class.

Here is how a professional Page Class looks: 

public class LoginPage {

    WebDriver driver;

    // 1. Define Locators using @FindBy
    @FindBy(id = "username")
    WebElement txtUsername;

    @FindBy(id = "password")
    WebElement txtPassword;

    @FindBy(xpath = "//button[@type='submit']")
    WebElement btnLogin;

    // 2. Initialize elements in Constructor
    public LoginPage(WebDriver driver) {
        this.driver = driver;
        // This "activates" the elements
        PageFactory.initElements(driver, this);
    }

    // 3. Create Action Methods (Business Logic)
    public void performLogin(String user, String pass) {
        txtUsername.sendKeys(user);
        txtPassword.sendKeys(pass);
        btnLogin.click();
    }
}

The Pro Tip: Lazy Initialization

By default, PageFactory.initElements(driver, this) initializes elements immediately. However, usually, we want AjaxElementLocatorFactory.

Why? Because modern apps (React/Angular) are slow. Elements don't appear instantly. If you use the standard init, you might get NoSuchElementException.

Using AjaxElementLocatorFactory implements a "Lazy Wait" (essentially an Implicit Wait specifically for these elements). 

public LoginPage(WebDriver driver) {
    this.driver = driver;
    
    // Waits up to 15 seconds for elements to appear before throwing error
    AjaxElementLocatorFactory factory = new AjaxElementLocatorFactory(driver, 15);
    
    PageFactory.initElements(factory, this);
}

Summary

  • Maintenance: If UI changes, you update only the Page Class, not the Test Class.
  • Readability: Tests become readable English (e.g., loginPage.performLogin()).
  • Stability: Use AjaxElementLocatorFactory to handle dynamic elements automatically.
